    Pre-Commit Quality Gates¶

    Overview¶

    The centralized-rules hook automatically detects git operations (commits, pushes, PRs) and enforces pre-commit quality gates to ensure code quality before changes are committed.

    How It Works¶

    Detection Mechanisms¶

    The hook detects git operations through two methods:

    1. Regular Keywords¶

    When your prompt contains git-related words: - commit - push - pull request / pr - merge - branch - rebase - cherry-pick - git add

    Example prompts: 

    "commit these changes"
"push to remote"
"create a pull request"
"merge this branch"

    2. Slash Commands¶

    When you invoke git-related skills via slash commands: - /xgit - /git - /xcommit - /commit - /push

    Example prompts: 

    "/xgit"
"/commit all changes"
"/push to origin"

    Quality Gates Triggered¶

    When a git operation is detected, the following pre-commit checks are automatically enforced:

    Required Checks (in order):¶

    1. Run Tests
    2. Execute all unit tests
    3. Ensure 100% pass rate

    4. Block commit if any test fails

    5. Security Scan

    6. Check for vulnerabilities
    7. Scan for secrets/credentials

    8. Verify secure coding practices

    9. Code Quality

    10. Run linters (ESLint, Pylint, etc.)
    11. Check code complexity

    12. Verify code standards compliance

    13. Refactoring Check

    14. Detect code smells
    15. Check for anti-patterns
    16. Identify improvement opportunities

    Workflow¶

    When you trigger a git operation, Claude will:

    1. Announce the pre-commit checks

    2. Execute each check in order

    Configuration¶

    Customizing Checks¶

    You can customize which checks run by modifying the hook script:

    Location: .claude/hooks/activate-rules.sh

    Section: generate_activation_instruction() function

    Example - Add additional check:

    REQUIRED CHECKS (run in this order):
  1ï¸âƒ£  Run tests        - Ensure all tests pass
  2ï¸âƒ£  Security scan    - Check for vulnerabilities
  3ï¸âƒ£  Code quality     - Verify code meets standards
  4ï¸âƒ£  Refactoring      - Check for code smells
  5ï¸âƒ£  Performance      - Run performance benchmarks  # NEW

    Disabling Quality Gates¶

    To temporarily disable quality gates, you can:

    1. Skip the hook entirely: 

      # Remove or rename the hook
mv .claude/hooks/activate-rules.sh .claude/hooks/activate-rules.sh.disabled

    2. Modify detection logic: Comment out the is_git_operation check in activate-rules.sh

    Testing¶

    Run Tests¶

    # Test slash command detection

    Expected Results¶

    • 16 slash command tests should pass
    • 17 quality gate tests should pass

    Benefits¶

    1. Consistency
    2. Same quality gates apply regardless of invocation method

    3. Keywords and slash commands trigger identical checks

    4. Safety

    5. Prevents committing broken code
    6. Catches security vulnerabilities early

    7. Enforces code quality standards

    8. Transparency

    9. User always knows what checks will run
    10. Clear feedback on pass/fail status

    11. Actionable error messages

    12. Flexibility

    13. Works with any git workflow
    14. Compatible with external skills
    15. Customizable check list

    Troubleshooting¶

    Quality Gates Not Triggering¶

    Problem: You type "commit changes" but don't see the quality gates banner

    Solutions: 1. Verify hook is installed: ls -la .claude/hooks/activate-rules.sh 2. Check hook is executable: chmod +x .claude/hooks/activate-rules.sh 3. Verify hook is configured in .claude/settings.json 4. Run test: echo '{"prompt":"commit"}' | .claude/hooks/activate-rules.sh

    Tests Failing¶

    Problem: Pre-commit quality gate tests fail

    Solutions: 1. Check hook syntax: bash -n .claude/hooks/activate-rules.sh 2. Verify regex patterns: Test individual prompts manually 3. Review test output for specific failures 4. Check for jq installation: which jq

    False Positives¶

    Problem: Quality gates trigger when they shouldn't

    Solutions: 1. Review keyword list in is_git_operation() function 2. Make regex patterns more specific 3. Add negative patterns to exclude certain cases 4. Adjust detection logic in activate-rules.sh
