    Context Management for AI Assistants¶

    Load when: AI responses are off-target or working with large codebases

    Provide Sufficient Context¶

    AI needs context to generate appropriate, correct code.

    Essential Context Elements¶

    Every AI request should include:

    • Purpose: What problem are we solving?
    • Constraints: Technical limitations, requirements
    • Existing patterns: How is similar code structured?
    • Dependencies: What libraries/frameworks are available?
    • Error conditions: What can go wrong?
    • Environment: Development, production, scale considerations

    Example: Poor vs. Good Context¶

    ❌ Poor: "Create a function to process payments"

✅ Good: "Create a process_payment function for our Django e-commerce app that:
- Accepts Payment object with amount, currency, payment_method
- Integrates with existing Stripe API wrapper (stripe_client)
- Returns PaymentResult with success/failure and transaction_id
- Raises PaymentError for invalid amounts or failed transactions
- Logs all payment attempts to payment_logger
- Follows existing service layer pattern in services/billing.py
- Must work in multi-server deployment with PostgreSQL"

    Reference Existing Code¶

    Point AI to concrete examples:

    Pattern Matching¶

    "Implement user deletion following the pattern in user_service.py

See delete_product() in product_service.py for our soft-delete pattern:
- Set deleted_at timestamp
- Keep record in database
- Filter deleted items in queries

Apply the same pattern to User model."

    Architecture Alignment¶

    "Add a new API endpoint following our existing structure:

Example from products API (src/api/products.py):
- Use @require_auth decorator
- Validate input with Pydantic model
- Handle errors with ErrorResponse
- Return JSON with status codes

Create similar endpoint for orders."

    Managing Token Limits¶

    Strategies for Large Codebases¶

    1. Be Selective
    2. Share only relevant files
    3. Don't dump entire codebase

    4. Focus on interfaces and contracts

    5. Use Summaries 

      "We have a ProductService with methods:
- create(name, price) -> Product
- find_by_id(id) -> Product
- delete(id) -> None

Create similar UserService with email-based lookup."

    6. Extract Interfaces 

      # Share signature, not implementation
class PaymentProcessor:
    def process(self, payment: Payment) -> PaymentResult:
        """Process payment and return result"""
        ...

    7. Break Into Chunks

    8. Handle one module at a time
    9. Complete and test each piece
    10. Build incrementally

    Context Window Optimization¶

    Instead of: 

    [Paste entire 500-line file]
"Add validation to line 247"

    Do this: 

    "In user_service.py, the create_user function (lines 245-260) needs
email validation. Current code:

def create_user(email: str, name: str):
    user = User(email=email, name=name)
    db.save(user)
    return user

Add email format validation before creating user."

    Session Continuity¶

    Maintain Context Across Conversation¶

    Good practices:

    1st message: "Working on user authentication for e-commerce app using Django + PostgreSQL"

2nd message: "Add login endpoint" (context maintained)

3rd message: "Add password reset" (still in auth context)

    Anti-pattern:

    1st message: "Create user model"
2nd message: "Add function to process payments" (context lost - how are these related?)
3rd message: "Fix the bug" (what bug? where?)

    Reset Context When Needed¶

    If switching domains:

    "I'm now working on a different feature - the order processing system.
Background: E-commerce app, Express.js backend, MongoDB database.
Previous auth discussion is complete."

    Providing Domain Knowledge¶

    Business Context¶

    "In our system, orders can be in these states:
- pending: Just created, awaiting payment
- paid: Payment confirmed
- shipped: Sent to customer
- delivered: Customer confirmed receipt
- cancelled: User or admin cancelled

Create a state machine for order transitions with validation rules:
- Can't ship before payment
- Can't cancel after shipping"

    Technical Context¶

    "Our deployment:
- 3 application servers behind load balancer
- PostgreSQL database with read replicas
- Redis for session storage and caching
- All state must survive server restarts

When implementing cart storage, use Redis with 24-hour TTL."

    Handling Ambiguity¶

    When requirements are unclear, AI makes assumptions. Be explicit:

    Specify Defaults¶

    ❌ "Add pagination to user list"

✅ "Add pagination to user list:
- Default page size: 20
- Max page size: 100
- 0-indexed pages
- Return total count in response"

    Define Error Handling¶

    ❌ "Add error handling"

✅ "Add error handling:
- Invalid input: return 400 with validation errors
- Not found: return 404 with error message
- Server errors: return 500, log to error_logger
- Follow error format in api/errors.py"

    Clarify Scale Requirements¶

    ❌ "Add search functionality"

✅ "Add search functionality:
- Dataset: ~100K users, growing 10K/month
- Search by name/email
- Response time: <200ms
- Use PostgreSQL full-text search (don't add Elasticsearch)"

    Context for Different Task Types¶

    New Feature¶

    "Adding feature: export user data to CSV

Context:
- User model fields: id, email, name, created_at, last_login
- Must comply with GDPR (include all user data)
- Users access via /api/users/export endpoint
- Authenticated users can only export their own data
- Admins can export any user data
- Use existing auth middleware (@require_auth)
- Follow CSV export pattern in reports/export.py"

    Bug Fix¶

    "Fixing bug: users can view deleted products

Current behavior:
- Product.deleted_at is set when deleting
- API still returns deleted products in searches

Expected behavior:
- Search should filter out products where deleted_at IS NOT NULL
- Direct lookup by ID should return 404 for deleted products

Location: api/products.py, search_products() function (lines 45-60)"

    Refactoring¶

    "Refactoring: extract email sending logic

Current state:
- Email code scattered across auth, orders, notifications
- Different SMTP configs in each place
- No retry logic
- Hard to test

Goal:
- Single EmailService class
- Centralized SMTP config
- Retry with exponential backoff
- Mock-friendly for tests

Follow service pattern in services/payment_service.py"

    Progressive Disclosure¶

    Start broad, get specific:

    Round 1: "I need to add user notifications"
AI: "What type of notifications? Email, SMS, push?"

Round 2: "Email notifications for order status changes"
AI: "What status changes should trigger emails?"

Round 3: "Send email when order moves to 'shipped' or 'delivered'"
AI: "Should we use existing EmailService?"

Round 4: "Yes, use EmailService. Template should include order ID, tracking number, estimated delivery"
AI: [Generates specific implementation]

    Summary: Context Best Practices¶

    1. Be Specific: Provide concrete requirements, not vague goals
    2. Show Examples: Reference existing code patterns
    3. Explain Constraints: Technical, business, scale limitations
    4. Define Success: What does "working" look like?
    5. Stay Focused: One task at a time, clear scope
    6. Maintain Session: Build on previous context
    7. Reset When Needed: Clear context for new domains
    8. Clarify Assumptions: Make implicit requirements explicit
