    AI Code Verification & Review¶

    Load when: Reviewing AI-generated code before committing

    Always Verify AI Output¶

    NEVER blindly accept AI code. Every output requires human review.

    Five-Point Check¶

    1. Read the code - Understand what it does
    2. Run tests - Automated verification
    3. Check edge cases - Test boundary conditions
    4. Review security - Validate inputs, check vulnerabilities
    5. Assess simplicity - Could it be simpler?

    Verification Checklist¶

    Correctness¶

    • [ ] Code compiles/runs without errors
    • [ ] All tests pass
    • [ ] Handles edge cases (null, empty, invalid input)
    • [ ] Logic is sound and correct
    • [ ] Returns expected results

    Security¶

    • [ ] Inputs are validated
    • [ ] No SQL injection vulnerabilities
    • [ ] No XSS vulnerabilities
    • [ ] Secrets not hardcoded
    • [ ] Authentication/authorization checked
    • [ ] Follows OWASP guidelines

    Quality¶

    • [ ] Follows project coding standards
    • [ ] Clear variable/function names
    • [ ] Includes necessary error handling
    • [ ] Performance is acceptable
    • [ ] Documentation/comments where needed

    Simplicity¶

    • [ ] Not over-engineered
    • [ ] Could it be simpler?
    • [ ] Necessary abstractions only
    • [ ] No premature optimization

    Common AI Security Mistakes¶

    1. SQL Injection¶

    # ❌ AI-generated vulnerability
def get_user(username):
    query = f"SELECT * FROM users WHERE username = '{username}'"
    return db.execute(query)

# ✅ Secure parameterized query
def get_user(username):
    query = "SELECT * FROM users WHERE username = ?"
    return db.execute(query, (username,))

    2. Missing Input Validation¶

    # ❌ No validation
def create_user(email, age):
    user = User(email=email, age=age)
    db.save(user)

# ✅ Proper validation
def create_user(email: str, age: int):
    if not is_valid_email(email):
        raise ValueError("Invalid email format")
    if age < 0 or age > 150:
        raise ValueError("Invalid age")

    user = User(email=email, age=age)
    db.save(user)

    3. Exposed Secrets¶

    # ❌ Hardcoded secrets
API_KEY = "sk_live_abc123xyz"

# ✅ Environment variables
API_KEY = os.environ.get("API_KEY")
if not API_KEY:
    raise RuntimeError("API_KEY not configured")

    4. Insecure Password Handling¶

    # ❌ Plain text passwords
def create_user(username, password):
    user = User(username=username, password=password)

# ✅ Hashed passwords
import bcrypt

def create_user(username, password):
    hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
    user = User(username=username, password_hash=hashed)

    Review for Over-Engineering¶

    Red Flags¶

    AI often suggests unnecessary complexity:

    # ❌ Over-engineered for simple config
class ConfigurationManagementSystemFactory:
    def create_loader(self) -> AbstractConfigurationLoader:
        return ConfigurationLoaderFactory().create(
            validator=ConfigValidatorFactory().create(),
            parser=ConfigParserFactory().create(),
            cache=ConfigCacheFactory().create()
        )

# ✅ Simple and direct
def load_config():
    with open('config.json') as f:
        return json.load(f)

    Questions to Ask¶

    • Do we really need this abstraction layer?
    • Is this design pattern necessary?
    • Could we solve this more directly?
    • Are we optimizing too early?
    • Is this harder to understand than it needs to be?

    Iterative Refinement¶

    Start working, refine gradually:

    # Iteration 1: Make it work
def search_users(query: str):
    users = User.query.all()  # Naive but working
    return [u for u in users if query.lower() in u.name.lower()]

# Iteration 2: Make it right
def search_users(query: str):
    return User.query.filter(
        User.name.ilike(f'%{query}%')
    ).limit(100).all()

# Iteration 3: Make it production-ready
def search_users(query: str, limit: int = 100):
    if not query or len(query) < 2:
        raise ValueError("Query must be at least 2 characters")

    return User.query.filter(
        User.name.ilike(f'%{query}%'),
        User.deleted_at.is_(None)
    ).order_by(User.name).limit(limit).all()

    Edge Case Testing¶

    Verify AI code handles:

    • Null/None values
    • Empty strings/lists
    • Boundary values (0, -1, max int)
    • Invalid input types
    • Concurrent access
    • Network failures (if applicable)
    • Disk full (if writing files)

    Performance Review¶

    Check for performance issues:

    • N+1 queries in database code
    • Inefficient algorithms (e.g., nested loops on large data)
    • Memory leaks (unclosed resources)
    • Blocking operations in async code

    Documentation Review¶

    Ensure AI code is documented:

    def calculate_compound_interest(
    principal: float,
    rate: float,
    time: int,
    compounds_per_year: int = 12
) -> float:
    """
    Calculate compound interest using A = P(1 + r/n)^(nt)

    Args:
        principal: Initial investment amount
        rate: Annual interest rate (as decimal, e.g., 0.05 for 5%)
        time: Investment period in years
        compounds_per_year: Number of times interest compounds per year

    Returns:
        Final amount including interest

    Example:
        >>> calculate_compound_interest(1000, 0.05, 10, 12)
        1647.01
    """
    return principal * (1 + rate / compounds_per_year) ** (compounds_per_year * time)

    When to Reject AI Code¶

    Reject if:

    • Security vulnerabilities present
    • Tests don't pass
    • Significantly more complex than needed
    • Doesn't follow project patterns
    • Missing critical error handling
    • Performance is unacceptable

    Better to reject and re-prompt than ship bad code.

    Commit Guidelines¶

    Only commit when:

    • [ ] All tests pass
    • [ ] Security review complete
    • [ ] Code reviewed for simplicity
    • [ ] Edge cases handled
    • [ ] Documentation adequate
    # Good commit message for AI-assisted code
git commit -m "Add user email validation

Implemented with AI assistance:
- Email format validation using regex
- Domain validation against allowed list
- Test coverage for edge cases

Reviewed for security and simplicity."
